## 0. 设计阶段“完成”的退出条件（我们要产出什么）

设计阶段可以认为完成，当且仅当我们能给出下面 4 个产物（有一个缺失就会继续陷入细节）：

1) **vNext control 的北极星目标 + v1 范围**（一页以内）
   - 明确 vNext 主要服务对象（前端/脚本/调试 CLI）、主要用例、明确非目标（例如：不靠 socat 手敲）。
   - 明确“哪些能力 v1 必须有；哪些明确延后（写进 Non-goals）”。
2) **单一真相的协议与命令面**（可实现、可测试）
   - `CONTROL_PROTOCOL_VNEXT.md` + `CONTROL_COMMANDS_VNEXT.md` 成为唯一规范；
   - 其它 `*_TMP.md` 仅保留“挑刺/备忘”，不再包含与规范冲突的契约。
3) **资源/状态模型（source-of-truth/所有权/落盘/一致性）**（这是现在最缺的）
   - 对每个资源（CONFIG / DOMAINRULES / DOMAINPOLICY / DOMAINLISTS / IPRULES / METRICS / STREAM）回答：
     - 谁写？谁读？谁是单一真相？
     - 哪些字段“由前端驱动但落盘在 daemon”（例如 subscription 元信息）？
     - 失败语义与一致性语义是什么（原子 apply、参照完整性、幂等性、并发写入）？
4) **迁移与下线策略（vNext 与 legacy 如何共存）**
   - vNext endpoint 形态/权限/gating；
   - “vNext 稳定”的判据与时间窗；
   - legacy 60606 的关闭/删除策略与回滚策略。

## 1. 已经完成/基本收敛的设计块（避免重复讨论）

- vNext wire：**netstring + JSON request/response envelope**（并要求结构化错误、严格拒绝歧义）。
- 不背 legacy token 语法：人类交互交给 `sucre-snort-ctl`（而非协议本身）。
- stream：control 不独占；限制的是“订阅拓扑”（每 streamType 单订阅、独立连接等）。
- 域名匹配/规则：原则上**尽量保持现实现不动**（包括 `type=domain` 仍按 regex 解释等已确认项）。

## 2. 设计阶段剩余的大块工作（按优先级）

### 2.1 北极星目标 + v1 scope 冻结（Stop rule）

为什么必须做：
- 没有“v1 只保证什么”的边界，就会把所有边角都当成必须钉死，讨论永不收敛。

需要回答的最小问题：
- vNext control 的 **核心成功指标**是什么？（例如：可解析、可解释、可回归、可灰度迁移）
- v1 明确不做哪些事？（例如：鉴权、HTTP 拉取、复杂跨资源事务、完美的人肉可用性）
- 细节进入规范的门槛是什么？（建议：只要影响兼容/安全/可解释性/实现可验证性才进入 v1）

建议输出：
- 新增一份 `Vnext_NorthStar_V1.md`（或直接在 `CONTROL_PROTOCOL_VNEXT.md` 顶部加一节）。

### 2.2 “资源/状态模型”补齐（当前最大缺口）

为什么它是最大缺口：
- 你刚澄清的订阅模型（前端驱动 refresh，但订阅字段落盘在 daemon）就是典型例子：  
  没有状态模型，“字段是否进入 daemon API”“谁维护语义”“并发写入如何仲裁”就会不断反复。

每个资源必须写死的内容（避免实现阶段各自猜）：
- 持久化：哪些字段必须落盘，哪些纯内存。
- 所有权：哪些字段由前端驱动写入、daemon 仅持久化；哪些字段由 daemon 自己维护。
- 并发：允许多 control client 吗？允许同时写同一资源吗？冲突怎么报？
- 一致性：跨资源引用（例如 policy 引 rules）是否做参照完整性；apply 是否原子；幂等语义。

建议输出：
- 在 `CONTROL_COMMANDS_VNEXT.md` 增加每个资源的 “State/Ownership/Consistency” 小节（每节 10 行以内）。

### 2.3 命令面“收口”与去重（规范文档只保留一个口径）

为什么必须做：
- 现在目录里同时存在“规范”“简化提案”“挑刺审阅”，如果不明确哪个是规范，后续一定自相矛盾。

必须做的收口动作：
- 明确：`CONTROL_COMMANDS_VNEXT.md` 是唯一命令规范；`CONTROL_VNEXT_SURFACE_SIMPLIFIED_TMP.md` 要么合并进规范、要么标注“已过期/仅供历史”。
- 在每个 `*_TMP.md` 顶部写清楚：它是“非规范”，不得被实现当作契约来源。

### 2.4 兼容/迁移：endpoint 并存、版本协商、灰度与回滚

为什么必须做：
- 你定义的“vNext 稳定判据”（两个小版本+无控制面反馈问题）属于迁移策略的一部分；若不落到文档，落地时会各自理解。

需要写死的最小集合：
- `HELLO` 能力协商（版本、`maxFrameBytes`、关键 limits、是否 strict-unknown）。
- vNext 与 legacy 并存窗口与“默认客户端”的切换步骤。
- legacy 60606 的关闭标准与回滚开关。

### 2.5 风险清单（仅保留会导致方向错误的“一级风险”）

我们不再收集小风险；只保留“如果错了会推翻方向”的：
- 是否真的需要 netstring？（你已接受 CLI -> 倾向稳定）
- control 是否必须支持多客户端并发写？（会影响状态模型与错误模型）
- `DOMAINLISTS` 订阅字段进 daemon API 后，是否会导致状态二义来源？（需要所有权与 patch/replace 语义）
- 域名 canonicalization 的“不做”会不会在产品层面造成不可接受的误解成本？（是否需要只做 trailing dot 这一类最小一致性）

## 3. “停止继续细化”的建议规则（防止再陷回细节）

从现在开始，任何新细则只有满足其一才进入 v1 规范，否则进入“实现备注/后置”：
- 影响兼容（会导致旧客户端/旧前端无法工作）
- 影响安全（会导致 silent bypass/策略误生效）
- 影响可解释性（无法给用户解释“为什么”）
- 影响可验证性（无法写测试/无法观测是否正确）
